# JobOS — Architecture Document

> **Purpose:** Detailed breakdown of modules, components, data flows, and dependencies. This is the source of truth for building and is referenced by Claude Code during development.
> **Related:** `jobos-design-document.md` (high-level overview), `jobos-visualization-guide.md` (HTML visualization instructions)

## System Overview

JobOS is structured as three layers:

1. **Foundation Layer** — shared context files and principles that every module reads
2. **Entry Gate** — single input mechanism for job opportunities
3. **Module Layer** — eight capability modules, each with its own agents, templates, and outputs

```
┌─────────────────────────────────────────────────────┐
│                   MODULE LAYER                       │
│  M1  M2  M3  M4  M5  M6  M7  M8                    │
│  ↕   ↕   ↕   ↕   ↕   ↕   ↕   ↕   ↕   ↕           │
├─────────────────────────────────────────────────────┤
│                   ENTRY GATE                         │
│           opportunity.md (per job)                   │
├─────────────────────────────────────────────────────┤
│                FOUNDATION LAYER                      │
│  profile.md | cv.md | target-roles.md | CLAUDE.md    │
└─────────────────────────────────────────────────────┘
```

## Foundation Layer

These are root-level context files that every module can read. They represent the user's identity and preferences.

### `profile.md`
**What it contains:** The user's professional identity — experience summary, skills, strengths, work style, what energizes vs. drains them, career narrative.
**Who fills it in:** The user, during initial setup.
**Who reads it:** All modules. This is the "who am I" file.
**Template sections (preliminary):**
- Professional summary
- Key skills and domains
- Work style and preferences
- Strengths (e.g., CliftonStrengths, self-assessment)
- What I'm looking for / what I want to avoid
- Career narrative (the story I tell about my trajectory)

### `cv.md`
**What it contains:** Structured CV content — the canonical, complete version of the user's professional history.
**Who fills it in:** Initially generated by parsing a LinkedIn PDF export, then reviewed and corrected by the user.
**Who reads it:** Module 3 (CV Optimization), Module 4 (Cover Letters), Module 5 (Homework Help), Module 6 (Interview Simulation).
**Format:** Structured markdown with clear sections: contact info, summary, experience (company, role, dates, achievements), education, skills, certifications.

### `target-roles.md`
**What it contains:** What the user is looking for — target role types, industries, company sizes, geographic preferences, salary expectations, must-haves, deal-breakers.
**Who fills it in:** The user, guided by Module 1 (Goal Setting).
**Who reads it:** Module 2 (Planning), Module 3 (CV tailoring), Module 4 (Cover Letters), Module 8 (Company Intelligence).

**Note on principles:** The two core principles (Honesty, Tough Love) are embedded directly in `CLAUDE.md`, which Claude Code loads automatically every session. They are also described in `README.md` for human readers. There is no separate principles file — this avoids an unnecessary dependency hop while ensuring every agent inherits the principles automatically.

## Entry Gate: Opportunity Input

Every opportunity enters through a single mechanism that normalizes data for all downstream modules. There are two entry paths.

### Flow
```
Path A: JD-first                    Path B: Company-first (Exploring)
─────────────────                   ────────────────────────────────
User provides URL or text           User wants to track a company
        ↓                                    ↓
   [Fetch & Parse]                  [Create minimal opportunity.md]
        ↓                                    ↓
   opportunity.md (full)            opportunity.md (company + context note)
        ↓                                    ↓
   Stage: Saved                     Stage: Exploring
        ↓                                    ↓
   Available to all modules         Available to all modules
                                    (JD-dependent modules gracefully stop)
                                             ↓
                                    [When JD arrives later]
                                             ↓
                                    Populate JD fields, rename folder,
                                    move to Saved → full flow
```

### `opportunity.md` — Normalized Format
Each opportunity file follows the same structure. Fields marked *(required)* must be present at creation; fields marked *(JD-available)* are populated when a JD becomes available and left empty with a comment at Exploring stage.

- Company name *(required)*
- Context note *(required at Exploring stage)* — why this company is being tracked, current engagement
- Role title *(JD-available)*
- Location / remote status *(JD-available)*
- Job description (full text, preserved) *(JD-available)*
- Key requirements extracted (skills, experience, qualifications) *(JD-available)*
- Nice-to-haves extracted *(JD-available)*
- Salary range (if stated) *(JD-available)*
- Application URL *(JD-available)*
- Date added *(required)*
- Current pipeline stage *(required)* — defaults to Exploring (company-first) or Saved (JD-first)

## Module Architecture — Designed (1-5)

### Module 1: Goal Setting

**Purpose:** Help the user define what they're looking for before applying anywhere.

**Components:**
- Agent: `goal-setting-advisor.md` — asks structured questions, challenges vague answers, pushes for specificity
- Output: populates `context/target-roles.md`

**Data flow:**
```
User input (conversation) → goal-setting agent → target-roles.md
```

**Dependencies:** None. This is a starting point.

**Key behaviors:**
- Asks about role type, industry, company size, culture, compensation, geography
- Challenges unrealistic combinations ("You want a startup salary at a Fortune 500 with full remote and no meetings — let's talk about tradeoffs")
- Outputs a clear, prioritized list of criteria

---

### Module 2: Pipeline & Planning

**Purpose:** Keep the job search moving forward. The module has two jobs: (1) capture what's happening across all opportunities with minimal user effort, and (2) advise the user on what to do next to progress toward their goals.

This is not a passive tracking table — it's an active planning advisor. The value isn't knowing where things stand; it's being told what to do about it.

**Components:**
- Agent: `planning-advisor.md` — scans all in-flight opportunities, assesses current state, recommends next actions
- File: `pipeline.md` — summary view of all opportunities and their stages (generated/maintained by the agent, not manually)
- File: `todo.md` — pending action items the user has committed to (maintained by the agent, reviewed each session)
- Files: `opportunities/*/activity-log.md` — per-opportunity activity history (one per opportunity folder)

**Stages:** Exploring → Saved → Researching → Applied → Screening → Interview → Homework → Offer → Accepted / Rejected / Withdrawn

**Exploring** is the pre-JD stage — covers recruiter outreach, networking, and company research before any specific role exists. Once a JD appears, the opportunity moves to Saved.

**Activity capture — two input modes:**

The planning agent is only useful if it knows what's actually happening. Activity capture must be near-zero effort for the user.

1. **User-initiated:** The user tells the system what happened in natural language at any point. Examples: "Had a screening call with Acme, went well, they want a second round next week." "Beta Inc rejected me." "Submitted the homework for Gamma." The agent parses this into a structured entry in the relevant `activity-log.md`.

2. **Agent-prompted:** At session start (or when the user asks for planning advice), the agent asks: "Anything new with your in-flight opportunities?" One question, natural language response. This lowers the bar from "remember to update your pipeline" to "just answer when asked."

Both modes feed into the same per-opportunity `activity-log.md` files.

**`activity-log.md` — per-opportunity activity history:**

Each opportunity folder contains an activity log. Entries are structured but simple — the agent handles formatting, the user never writes these directly.

```markdown
## Activity Log: Acme Corp — Senior PM

| Date | Activity |
|------|----------|
| 2026-03-10 | Saved opportunity. Strong match for target criteria — remote, Series B, PM leadership role. |
| 2026-03-11 | Tailored CV — emphasized platform migration experience and team scaling metrics to match JD. |
| 2026-03-12 | Submitted application via company careers page. Expect response within ~2 weeks. |
| 2026-03-18 | Screening call with recruiter. Positive — discussed team structure, they want a second round next week. Expect scheduling confirmation by Friday. |
```

Activity descriptions are as long as needed to capture the full context — stage implications, next expected actions, outcomes, and relevant details are all written naturally into the description. Stage tracking (current stage per opportunity) lives in `pipeline.md`, not in the activity log. The activity log is the history — what happened and when.

Any module that performs work on a specific opportunity appends to that opportunity's `activity-log.md`. The planning advisor is the only module that reads across all logs to make recommendations.

**Planning recommendations:**

The planning agent reads across all opportunity folders (activity logs, opportunity.md files, and any other module outputs present) and cross-references with `target-roles.md` to produce actionable recommendations. These span both in-system actions and real-world follow-ups.

Example recommendation types (illustrative, not exhaustive):
- "You saved 3 opportunities last week but haven't applied to any. Which ones do you want to move forward on?"
- "Your interview with Acme was 5 days ago and you haven't uploaded the transcript. Want to run interview analysis (Module 7)?"
- "Beta Inc was supposed to get back to you by Wednesday. Consider sending a follow-up."
- "You have a homework assignment due for Gamma in 3 days. Want to start working on it (Module 5)?"
- "You've been researching for two weeks without applying anywhere. Time to commit or refine your target criteria."
- "Delta Corp hasn't progressed past 'Applied' in 3 weeks. Flag as stale or follow up?"

**Todo list — persistent action tracking:**

The planning advisor recommends actions, but recommendations are lost between sessions. `todo.md` captures what the user has explicitly committed to doing. Only the planning advisor writes to this file.

Session flow:
1. Read `todo.md` — review pending items with the user (done, still pending, or no longer relevant)
2. Check-in — "Anything new with your in-flight opportunities?"
3. Scan and recommend — produce prioritized recommendations based on current state
4. Commit — ask which recommendations the user wants to add to the todo list

Completed items are removed from `todo.md` — the activity logs already capture what was done. The todo list only tracks what's pending. If an item persists across multiple sessions, the agent escalates its tone.

**Data flow:**
```
(Session start)
   Reads todo.md → reviews pending items with user
        ↓
   Asks for activity updates (natural language)
        ↓
   planning-advisor agent
        ↓
   Parses → writes to relevant activity-log.md
   Updates → pipeline.md (summary view)
        ↓
   Reads all activity-log.md files + opportunity.md files + module outputs
        ↓
   Cross-references with target-roles.md
        ↓
   Produces prioritized recommendations
        ↓
   User commits to specific actions → writes to todo.md
```

**Dependencies:**
- Reads: `context/target-roles.md`, `todo.md`, all `opportunities/*/activity-log.md`, all `opportunities/*/opportunity.md`, and presence/absence of other module outputs (cv-variant.md, company-research.md, etc.)
- Writes: `opportunities/*/activity-log.md`, `pipeline.md`, `todo.md`
- Awareness of: all other module workflows (knows what comes after what, what's missing, what's overdue)

**Design notes:**
- The planning agent needs to understand the workflows of other modules to make good recommendations — e.g., it should know that after an interview, transcript upload and analysis is the expected next step.
- `pipeline.md` still exists as a summary view, but it's maintained by the agent, not the user. It's a generated snapshot, not a manually updated table.
- The planning quality is directly proportional to how complete the activity data is. The interaction model is designed to make sharing activity nearly zero effort. If it ever feels like admin work, the user stops doing it and the module breaks.
- Recommendations should be opinionated and direct (Principle 2). "You might want to consider following up" is weak. "Beta Inc was supposed to respond 4 days ago. Send a follow-up or mark it as stale." is useful.

### Module 3: CV Optimization

**Purpose:** Transform raw CV content into a polished, tailored document.

**Components:**
- Agent: `cv-reviewer.md` — one-time base CV review (structure, wording, gaps, emphasis)
- Agent: `cv-tailor.md` — per-opportunity tailoring given a specific JD
- Agent: `cv-feedback-recruiter.md` — reads a CV (base or variant) and reacts as a recruiter screening candidates
- Agent: `cv-feedback-cpo.md` — reads a CV (base or variant) and reacts as a CPO/VP Product evaluating a candidate
- Script/workflow: LinkedIn PDF parser → `cv.md`
- Template: `cv-pdf-template.html` — HTML template for PDF generation
- Script/workflow: `cv-variant.md` → HTML → PDF

**Data flow:**
```
LinkedIn PDF export
        ↓
   [Parse & Structure]
        ↓
   context/cv.md (base CV)
        ↓
   cv-reviewer agent → [completeness check → ask user for missing detail if needed]
                     → suggestions → user updates cv.md
        ↓
   (optional, user-triggered)
   cv.md → cv-feedback-recruiter + cv-feedback-cpo → persona feedback
        ↓
   User reviews feedback, updates cv.md if needed
        ↓
   (per opportunity)
   cv.md + opportunity.md → cv-tailor agent → cv-variant.md
        ↓
   (optional, user-triggered)
   cv-variant.md → cv-feedback-recruiter + cv-feedback-cpo → persona feedback
        ↓
   cv-variant.md → [HTML template] → [PDF generator] → final CV PDF
```

**Proactive suggestion rule:** Whenever `cv.md` or a `cv-variant.md` is updated, Claude Code suggests running the feedback personas. The user decides whether to run it — it's a nudge, not a gate. This rule lives in `CLAUDE.md` as a behavioral default.

**Dependencies:**
- Reads: `context/cv.md`, `context/profile.md`, `opportunities/*/opportunity.md`
- Writes: `opportunities/*/cv-variant.md`, final PDF

**Key behaviors (cv-reviewer):**
- **Completeness assessment first:** Before making any recommendations, scans each position for sufficient detail. If a role has only a title and dates, or lists only generic responsibilities with no achievements/metrics/outcomes, the reviewer stops and flags it: "I don't have enough detail about your role at [Company] to assess or improve it." It then asks structured questions — what were you accountable for, what did you ship, what changed because of your work, what would your manager say you did well — and only proceeds to recommendations once the gaps are filled. This is a prerequisite step, not optional. (Principle 1: better to ask than to invent.)
- Analyzes structure, ordering, language quality
- Identifies weak points: vague achievements, missing metrics, unclear impact
- Checks for dual-audience readability (recruiter keywords + PM craft signals)
- Suggests improvements — does NOT auto-apply them

**Key behaviors (cv-tailor):**
- Reads the specific JD requirements
- Identifies which parts of cv.md to emphasize, reorder, or reword
- Flags gaps honestly: "JD requires X, which isn't in your CV"
- Produces a complete cv-variant.md ready for PDF generation

**Key behaviors (cv-feedback-recruiter):**
- Reads the CV as a recruiter screening candidates for a PM role
- Output: what caught their attention, what's missing or unclear, whether they'd move the candidate forward, and why
- Checks for: keyword coverage, title/experience progression, readability, red flags (gaps, job-hopping, vague descriptions)
- Short, opinionated output — not a lengthy report

**Key behaviors (cv-feedback-cpo):**
- Reads the CV as a CPO/VP Product evaluating a candidate
- Output: what signals product craft, what feels generic or unconvincing, whether they'd want to interview this person, and why
- Checks for: outcome-oriented framing, product thinking signals, strategic vs. tactical balance, evidence of real impact
- Short, opinionated output — not a lengthy report
- **Design note:** This persona is harder to calibrate than the recruiter. The agent instructions will need careful examples of what good vs. bad looks like from a CPO's perspective. Expect iteration.

**Module 6 merge-later note:** The recruiter and CPO personas here are deliberately scoped to CV feedback only — they read a document and react to it. Module 6 (Interview Simulation) will need full interactive personas that conduct conversations, adapt to answers, and challenge reasoning. When Module 6 is designed, we'll evaluate whether to extend these personas or build interview-specific ones from scratch. The agent file structure supports either path.

### Module 4: Cover Letter Writing

**Purpose:** Generate a personalized cover letter when needed for a specific opportunity.

**Components:**
- Agent: `cover-letter-writer.md`
- Template: `cover-letter-template.md`

**Data flow:**
```
context/cv.md + context/profile.md + opportunity.md
        ↓
   cover-letter-writer agent
        ↓
   opportunities/*/cover-letter.md
```

**Dependencies:**
- Reads: `context/cv.md`, `context/profile.md`, `opportunities/*/opportunity.md`
- Optionally reads: `opportunities/*/company-research.md` (if Module 8 has run)
- Writes: `opportunities/*/cover-letter.md`

**Key behaviors:**
- Connects user's actual experience to the specific role
- Uses company context if available (from Module 8)
- Does NOT oversell or fabricate (Principle 1)
- Tone: professional, specific, authentic — not generic or formulaic

### Module 5: Homework Assignment Help

**Purpose:** Help the user complete take-home tasks and case studies.

**Components:**
- Agent: `homework-helper.md`

**Data flow:**
```
User provides assignment brief + context/profile.md + context/cv.md
        ↓
   homework-helper agent
        ↓
   Structured response / deliverable
```

**Dependencies:**
- Reads: `context/profile.md`, `context/cv.md`, assignment brief (user-provided)
- Optionally reads: `opportunities/*/opportunity.md`, `opportunities/*/company-research.md`

**Key behaviors:**
- Helps structure thinking, not writes the answer for the user
- Uses the user's actual knowledge and experience as the foundation
- If the assignment asks about something the user doesn't know, says so and helps them learn rather than fabricating expertise
- Output must survive a follow-up conversation about it (Principle 1 test)

## Module Architecture — TBD (6-7) and Designed (8)

Modules 6 and 7 have clear intent but need detailed design work before building. Module 8 is fully designed. What follows covers all three.

### Module 6: Interview Simulation ⚠️ TBD

**Intent:** Agents role-play as specific interview personas to give the user realistic practice.

**Planned personas:**
- VP of Product — tests product thinking, strategic reasoning, craft
- Headhunter / Recruiter — tests communication, motivation narrative, red flag avoidance
- Hiring Manager — tests team fit, collaboration style, specific domain knowledge

**What we know:**
- Should use opportunity.md + company research for realistic, contextual scenarios
- Must be challenging (Principle 2) — not a confidence-building exercise
- Should improve over time based on real interview feedback (Module 7)

**What we don't know yet:**
- Does the user type responses or speak them? (UX question with major design implications)
- How many rounds per persona? One session or iterative?
- How to calibrate difficulty — too easy builds false confidence, too hard is demoralizing
- How to prevent the "uncanny valley" problem where practice feels nothing like the real thing
- Feedback format: immediate after each answer, or consolidated at the end?

**Risk:** This is the hardest module to get right. A bad simulation is worse than no simulation because it builds false confidence. Needs careful design.

---

### Module 7: Interview Analysis & Feedback Loop ⚠️ TBD

**Intent:** User provides transcripts from real interviews. System analyzes performance and feeds learnings into future preparation.

**What we know:**
- Input: interview transcript (user records and provides)
- Output: analysis of what went well, what didn't, specific recommendations
- Should identify patterns across multiple interviews
- Findings should feed back into Module 6 (simulation targets real weaknesses)

**What we don't know yet:**
- Transcript format — what structure works best for analysis?
- How to measure "what went well" vs "what went poorly" without the outcome context (did the user advance?)
- How does the feedback loop mechanically work? A file that accumulates learnings? A running improvement tracker?
- How to avoid reinforcing bad patterns if the user keeps practicing the wrong things

### Module 8: Company Intelligence — Designed

**Purpose:** Produce a comprehensive company intelligence report covering business health, product analysis, and employee sentiment. One trigger, one structured output. Formerly three separate modules (8: Company Intelligence, 9: Employee Sentiment, 10: Product Deep Dive) — merged because the user benefits from a single research pass, not three separate triggers.

**Note:** Interview question generation (formerly part of Module 10) has moved to Module 6 (Interview Simulation), which will read `company-research.md` as input.

**Components:**
- Agent: `company-research-agent.md` — single agent file with internal separation by research area
- Template: `company-research-template.md` — structured output template with all sections always present

**Data flow:**
```
User triggers: "research [company]" / "tell me about [company]"
        ↓
   [Folder resolution — find or create opportunity folder]
        ↓
   company-research-agent
        ↓
   (Write-as-you-go, section by section)
        ↓
   1. Company Overview → web search: about, website, Crunchbase, LinkedIn
   2. Business Health → web search: funding, revenue, layoffs, news
   3. Product Analysis → web search: G2/Capterra reviews, blog, launches
   4. Employee Sentiment → web search: Glassdoor, Atmoskop, "working at"
   5. Role Fit Assessment → reads target-roles.md (skipped if unavailable)
   6. Sources and Limitations → compiled from all searches
        ↓
   opportunities/[folder]/company-research.md
        ↓
   Activity log entry appended
```

**Dependencies:**
- Reads: `context/target-roles.md` (for Role Fit Assessment), `opportunities/*/opportunity.md` (if exists)
- Writes: `opportunities/*/company-research.md`, appends to `opportunities/*/activity-log.md`
- Read by: Module 4 (Cover Letters), Module 5 (Homework), Module 6 (Interview Simulation — when built)

**Key behaviors:**
- **Write-as-you-go:** Research and write each section sequentially. Preserves partial output if interrupted and keeps context window manageable.
- **Per-section confidence:** Every major section (Business Health, Product Analysis, Employee Sentiment) gets a High/Medium/Low confidence indicator with stated basis.
- **Facts first, opinion second:** Factual findings first, then editorial interpretation prefixed with "*Read:*". Keeps data and interpretation clearly separated.
- **Honest about gaps:** If minimal public data exists, says so directly. Does not pad thin sections with generic industry observations (Principle 1).
- **Staleness mechanism:** `Last researched: YYYY-MM-DD` in file header. Any module reading company-research.md older than 3 weeks flags it for refresh. Lightweight refresh updates changed sections; full re-run regenerates from scratch.
- **Future tool support:** Agent checks for Tavily/Firecrawl availability at start. Uses if configured, falls back to web search. No hardcoded MCP integrations.
- **Role Fit scope:** Company-level fit against target-roles.md only (industry, size, location, deal-breakers). Not role-specific skill matching (cv-tailor) or apply/don't-apply decisions.

**Refactor path:** If the agent file grows too complex, it can be split into sub-agents (business-health, product-analysis, employee-sentiment) that the main agent orchestrates. Single file is sufficient for now.

## Shared Components

### Agent Instruction Hierarchy

The project uses a two-tier instruction hierarchy:

```
CLAUDE.md (project orientation + behavioral defaults — auto-loaded by Claude Code)
    ↓
agents/<module-agent>.md (module-specific instructions)
```

**`CLAUDE.md`** — Read automatically by Claude Code at session start. Contains everything that applies to every session: what this project is, how the folder structure works, the two core principles (Honesty + Tough Love) written as agent instructions, pointers to context files (profile.md, cv.md, target-roles.md), tone expectations, gap-flagging behavior ("when you find a gap between the user's experience and a JD requirement, flag it transparently"), output quality standards, and guidance on which agent files to load for which tasks. Because Claude Code loads this automatically, every session starts with full project context — no extra files to remember to load.

**`agents/<module-agent>.md`** — Module-specific instructions only. Each follows the same structure:
1. Role definition (who is this agent?)
2. Input files it reads
3. Output files it writes
4. Specific behavioral rules for this module
5. Examples of good vs bad output

Module agents don't repeat principles, tone rules, or context file references — CLAUDE.md already covers all of that.

**Design rationale:** The user's natural language request is the routing mechanism — no dispatcher or master agent needed. When the user says "tailor my CV for this role," Claude Code already has CLAUDE.md loaded and reads the relevant agent file. Each agent file stays small and focused, which means better output quality (less noise in the prompt) and easier debugging. Merging shared instructions into CLAUDE.md avoids an extra dependency hop — every agent would have needed to reference shared-instructions.md separately, creating a point of failure for no practical benefit.

### PDF Generation Pipeline
Used by Module 3, potentially by Module 4:
```
markdown content → HTML (using template) → PDF (via scripts/generate-cv-pdf.py, headless Chromium)
```
Single template. Clean typography. Professional appearance.

## Technical Notes

- **Runtime:** Claude Code (CLI or via Cursor/IDE)
- **Storage:** GitHub repository (private for personal use, public for open source)
- **File format:** All content in markdown. HTML only for PDF template.
- **Web access:** Required for Entry Gate (fetching JD URLs), Module 8 (company research). Not required for Modules 1-5.
- **No external dependencies:** No databases, no APIs beyond Claude and web access. Everything runs from markdown files.
